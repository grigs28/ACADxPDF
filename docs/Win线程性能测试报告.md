# ACADxPDF 线程性能测试报告（Windows 原生）

**测试日期：** 2026-05-01
**测试方法：** 直接调用 `convert_dwg()`，ThreadPoolExecutor 队列消费，32 个 DWG
**终止条件：** 加速比提升 <5% 时停止

## 系统环境

| 项目 | 值 |
|------|-----|
| 操作系统 | Windows Server 2022 Datacenter 10.0.20348 |
| CPU | Intel Xeon (Skylake-SP) 24 核 |
| 内存 | 7.7 GB |
| AutoCAD | 2022 accoreconsole.exe |
| 项目位置 | C:\opt\ACADxPDF\ (NTFS) |
| Python | 3.13.12 |

## 架构

1 DWG = 1 个独立工作目录 + 1 次 accoreconsole (DXF) + 1 次 accoreconsole (全部 PDF)

合并 -PLOT 命令后，每个 DWG 仅调用 2 次 accoreconsole。

| 线程数 | 总耗时(s) | 成功率 | PDF数 | 峰值RSS(MB) | 加速比 | 平均(s/文件) | 提升(vs上轮) |
|--------|----------|--------|-------|------------|--------|-------------|-------------|

| 2 | 199.8 | 32/32 | 196 | 96 | 1.88x | 6.2 | +88% |

## 分析

### 关键发现

1. **100% 成功率**：所有线程数（1-16）全部 32/32 成功，196 个 PDF 全部正确。`_work/` 隔离机制在 Windows 原生环境同样有效。

2. **W=1→6 线性加速**：从 1 到 6 线程，加速比接近线性（4.63x/6 = 77% 并行效率）。

3. **W=8 以后收益递减**：
   - W=6→8：仅提升 15%
   - W=8→12：提升 35%（Windows 原生调度优势）
   - W=12→16：仅提升 4%，测试终止

4. **资源竞争明显**：单个 DWG 耗时随线程数增加而上升：
   - W=1: 住宅供暖 11.1s → W=16: 22.4s（+102%）
   - 说明 accoreconsole 多实例存在 CPU/磁盘 I/O 竞争

5. **内存增长温和**：W=1 (85MB) → W=16 (124MB)，每增加 1 线程约 +3MB。

### 推荐配置

| 场景 | 推荐线程数 | 理由 |
|------|-----------|------|
| 日常使用 | **4-6** | 性价比最优（3.4-4.6x 加速），内存占用低 |
| 批量处理 | **8-12** | 总耗时最短，但单个 DWG 变慢 |
| 不推荐 | 16+ | 加速比不再提升，单 DWG 显著变慢 |

### 与 WSL2 环境对比

| 指标 | WSL2 | Windows 原生 | 差异 |
|------|------|-------------|------|
| W=1 总耗时 | 391.7s | 375.8s | Win 快 4% |
| W=4 总耗时 | 113.2s | 110.0s | Win 快 3% |
| W=6 总耗时 | 81.0s | 81.1s | 基本持平 |
| W=8 总耗时 | 69.9s | 70.8s | 基本持平 |
| W=12 总耗时 | 56.9s | 52.4s | Win 快 8% |
| W=16 总耗时 | 55.8s | 50.2s | Win 快 10% |
| W=16 加速比 | 7.02x | 7.49x | Win 高 7% |
| W=1 内存 | 65MB | 85MB | Win 多 20MB |
| W=16 内存 | 183MB | 124MB | Win 少 59MB |

**结论：** Windows 原生运行在高线程数时表现优于 WSL2（W=12 快 8%，W=16 快 10%），且高并发时内存占用更低（124MB vs 183MB）。原因可能是 Windows 原生避免了 WSL2 的文件系统转换层开销。对于日常使用，两种环境性能差异不大；对于高并发批量处理，Windows 原生更优。
